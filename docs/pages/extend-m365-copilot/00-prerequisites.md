## Key concepts before you build

The bundles use five building blocks that are easy to confuse when you're new to them. Read this section once — it will save hours of "wait, what is that doing?" during the labs.

<div data-widget="concepts"
     data-cards="Microsoft 365 Copilot::blue::The AI host::The underlying LLM and orchestration layer. It decides when to call your agent, routes the conversation, and renders the response. You don't build this — you extend it.||Declarative Agent::purple::Your focused Copilot persona::A set of three JSON/text files that tell Copilot to behave as a specific specialist — with a name, custom instructions, and optionally tools it can call. Copilot provides the AI; you provide the purpose.||MCP — Model Context Protocol::teal::The tool contract::An open protocol (Anthropic, 2024) for connecting AI agents to external data and actions. An MCP server exposes named functions with typed schemas. The agent calls them by name; the server runs them and returns results.||Dev Tunnel::amber::Local → public HTTPS::A VS Code feature (backed by Azure) that gives your localhost a temporary public HTTPS URL. Needed because Microsoft 365 Copilot is a cloud service and can't reach &lt;code&gt;localhost:3001&lt;/code&gt; directly.||Azurite::coral::Local Azure Storage emulator::A Node.js process that pretends to be Azure Table Storage on your machine. The MCP server in Bundle A stores claims data in it — no Azure subscription or cost needed for local dev.||ai-plugin.json::green::The tool manifest::The file that tells a Declarative Agent which MCP tools exist, what their parameters are, and where the MCP server lives. Generated by Agents Toolkit; you'll read and edit it in Bundle A."></div>

Here's how all five relate at runtime in Bundle A:

<div data-widget="arch"
     data-rows="row::Microsoft 365 Copilot::copilot::Receives user query → decides to call your agent|Declarative Agent::agent::declarativeAgent.json + instruction.txt + ai-plugin.json||label::reads ai-plugin.json → calls MCP tool via HTTPS||row::Dev Tunnel::tunnel::Public HTTPS ↔ your localhost:3001|MCP Server (Node.js)::mcp::Handles tool calls · runs on your machine|Azurite (local DB)::data::Azure Table Storage emulator · port 10002"></div>

<div data-widget="callout"

## Exercise 1: Configure your M365 tenant

By default, Teams doesn't allow uploading custom apps — an admin toggle must be flipped first. If you did this in Lab E0 already, skip to Exercise 2.

<div data-widget="step" data-n="1" data-title="Enable custom app uploads in Teams"></div>

1. Go to [admin.microsoft.com](https://admin.microsoft.com) and sign in with your M365 admin account
2. In the left nav, click **Show all** → **Teams** (opens the Teams Admin Center)
3. In Teams Admin Center: **Teams apps** → **Setup policies**
4. Select **Global (Org-wide default)**
5. Toggle **Upload custom apps** to **On**
6. Click **Save** at the bottom

<cc-end-step lab="e0" exercise="1" step="1" />

<div data-widget="callout"
     data-type="warn"
     data-title="Propagation delay"
     data-body="This setting can take up to 24 hours to take effect in some tenants, though it's usually faster. If Provision fails with an upload error later, wait an hour and retry."></div>

<div data-widget="step" data-n="2" data-title="Verify your Copilot licence"></div>

Go to [m365.cloud.microsoft/chat/](https://m365.cloud.microsoft/chat/). If you see the Copilot Chat interface, you have a valid licence. If you see a licence error, contact your tenant admin — you need a Microsoft 365 Copilot licence assigned to your account.

## Exercise 4: Create your first Dev Tunnel

A Dev Tunnel solves a fundamental problem: Copilot is a cloud service, but during development your MCP server runs on your laptop. The tunnel punches a hole through NAT/firewall and gives your localhost a stable HTTPS URL that Copilot can reach.

<div data-widget="callout"
     data-type="concept"
     data-title="The analogy"
     data-body="Imagine you've built a vending machine in your garage. You want customers (Copilot) to use it, but they can't come to your garage. A Dev Tunnel is like giving your garage a real street address and putting up a transparent forwarding service — customers knock on the street address, and the knock arrives at your garage door."></div>

<div data-widget="step" data-n="1" data-title="Forward port 3001"></div>

1. In VS Code, open the integrated terminal panel
2. Click the **Ports** tab (next to Terminal)
3. Click **Forward a Port**
4. Enter `3001` and press Enter

<cc-end-step lab="e0" exercise="4" step="1" />

<div data-widget="step" data-n="2" data-title="Make it public"></div>

1. Right-click the new entry in the Ports list
2. Select **Port Visibility** → **Public**
3. If prompted, sign in with your GitHub account (required once)
4. Copy the forwarded address — it looks like: `https://abc123def456.use.devtunnels.ms`

<cc-end-step lab="e0" exercise="4" step="2" />

<div data-widget="step" data-n="3" data-title="Understand what &quot;public&quot; means"></div>

Marking the tunnel as Public means anyone with the URL can reach your local server without needing to authenticate — important because the Copilot service (running in Microsoft's cloud) needs to call your MCP endpoints without a login prompt. Your MCP server in Bundle A enforces its own security (and in E10, OAuth 2.0), so public visibility on the tunnel itself is safe for development.

<cc-end-step lab="e0" exercise="4" step="3" />

<div data-widget="callout"
     data-type="warn"
     data-title="Tunnels expire"
     data-body="Dev Tunnels reset when VS Code restarts. You'll need to recreate the tunnel and update the URL in your config files each new session. In Bundle A, this URL goes into &lt;code&gt;package.json&lt;/code&gt;'s inspector script and into your declarative agent's MCP server config."></div>

## Exercise 5: Understand MCP hands-on

Before building a full MCP server in Bundle A, let's look at what the protocol actually exposes. This makes the "15 tools in Zava's claims system" far less abstract.

<div data-widget="step" data-n="1" data-title="What an MCP server exposes"></div>

An MCP server has three kinds of objects. You'll encounter all three in Bundle A:

- **Tools** — callable functions with named parameters and typed outputs. The agent calls these by name (e.g. `get_claims`, `create_inspection`). This is the most important object type.
- **Resources** — readable data sources (files, database views) the agent can fetch. Less common in these labs.
- **Prompts** — predefined prompt templates the server exposes. Used in Lab E11's widget flow.

<cc-end-step lab="e0" exercise="5" step="1" />

<div data-widget="step" data-n="2" data-title="Read an MCP tool schema (no server needed)"></div>

Every tool in an MCP server has a JSON schema that describes its inputs and output. Here's what the `get_claim` tool from Bundle A looks like:

```json
{
  "name": "get_claim",
  "description": "Get details of a specific insurance claim by claim number",
  "inputSchema": {
    "type": "object",
    "properties": {
      "claimNumber": {
        "type": "string",
        "description": "The claim number, e.g. CN202504990"
      }
    },
    "required": ["claimNumber"]
  }
}
```

When a user asks "show me claim CN202504990", the LLM reads this schema, decides `get_claim` is the right tool, fills in `claimNumber`, and sends that to your MCP server. Your server runs the lookup and returns JSON. The LLM formats the result. You never write the routing logic — the protocol handles it.

<cc-end-step lab="e0" exercise="5" step="2" />

<div data-widget="step" data-n="3" data-title="Understand the ai-plugin.json bridge"></div>

The Declarative Agent doesn't talk to the MCP server directly at config time — it reads `ai-plugin.json`, generated by Agents Toolkit when you run "Fetch action from MCP." Here's the structure you'll see:

<div data-widget="tree"
     data-lines="appPackage/|├── declarativeAgent.json   [dim:# lists ai-plugin.json under &quot;actions&quot;]|├── [hi:ai-plugin.json]          [dim:# generated · lists all tool schemas + MCP server URL]|├── instruction.txt|└── manifest.json"></div>

<div data-widget="callout"
     data-type="info"
     data-title="Why this matters for Bundle A"
     data-body="When you &quot;Fetch action from MCP&quot; in Exercise 5 of Lab E8, Agents Toolkit calls your running MCP server, reads its tool list, and writes that into &lt;code&gt;ai-plugin.json&lt;/code&gt;. If you add or change tools on the server later, you re-run &quot;Fetch&quot; to regenerate the file. The agent reads the file — not the live server — to know what tools are available."></div>
